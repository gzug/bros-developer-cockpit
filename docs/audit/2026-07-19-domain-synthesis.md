# Developer Cockpit multi-domain discovery synthesis

- **Audit date:** 2026-07-19
- **Baseline:** `origin/main` at `55b3b019746f73ada23933146c1fedb60e90bbc4`
- **Brief:** [gzug/bros-developer-cockpit#64](https://github.com/gzug/bros-developer-cockpit/issues/64)
- **Scope:** Read-only discovery. This artifact changes no application code, runtime behavior, protected boundary, workflow, schema, route, or UI surface.

## Decision summary

The candidate domain list was retained unchanged. The repository contains eleven authenticated product screens plus distinct assistant, auth/path, shipping, persistence, test/eval, accessibility, documentation, and build layers. Merging any two would have hidden a real ownership boundary; splitting further would have exceeded the nine-domain cap without improving prioritization.

Exactly one read-only subagent reviewed each domain. All reviewers used the same inclusion lens: a finding had to affect end-user complexity, project-overview complexity, or future optimization work. Every retained finding is grounded in a current file and line; explicit clean/no-evidence results are retained where they close an important question.

The orchestrator permits at most six active subagents, so the nine reviews ran as a rolling `6 + 3` parallel wave rather than nine simultaneous processes. Domains were neither merged nor duplicated. The Projektleiter inventoried the repository first, wrote the questions, checked the returned evidence, and authored the cross-domain synthesis and ranking below.

The optional `data/audit/domain-findings.jsonl` was skipped. There is no current audit-data consumer or schema convention, and adding one would create a second deliverable contract without improving this decision baseline.

Protected-boundary note: findings may describe complexity around pause, approval, auth, paths, guardrails, workflows, environment, schemas, or the skills radar, but the simplification angles and ranked proposals deliberately target adjacent read models, caller contracts, tests, documentation, UI state handling, and build organization. They do not propose changing the protected boundaries themselves.

## Cross-domain synthesis

### One lifecycle is projected as several different truths

Product/IA, Shipping, Assistant, Auth, Docs, and Tests all point at the same root complexity: existing durable states are reinterpreted independently by route copy, timelines, owner actions, App Help, migration diagrams, and tests. “Approved” can become Checked or Ready; “Paused” can mean global availability or an item blocker; “Done” can mean closed, categorized completion, or deleted/not-planned; owner/Don/Main Dev and Control/Ideas also drift. This is not primarily a copy problem. It is a missing, canonical read projection over existing protected state.

### Unknown and failure frequently become plausible-looking data

Product/IA, Shipping, Data, Tests, and Accessibility converge on a second pattern: query failure becomes empty Plan/Done lists, unresolved connection data becomes disconnected warnings, missing Skills data becomes sample/default content, malformed snapshots disappear, partial imports still score, and DB/GitHub error behavior is weakly exercised. The system often prefers a displayable answer over an honest unknown. That directly violates the product's honesty law and makes operational diagnosis harder.

### Important boundaries exist, but as caller conventions

Auth, Shipping, Assistant, and Tests show that many individual helpers are careful while end-to-end guarantees remain distributed across callers. Owner enforcement is strong for most destructive handlers, but role capability intent is not captured as one matrix. Shipping writers rely on callers for issue ownership. Model paths use different honesty filters. Golden evaluator helpers and production engine parsing are parallel implementations. Safety therefore requires code archaeology rather than one executable contract.

### Product knowledge has no single freshness path

Assistant, Docs, Product/IA, Data, and Tests identify several manually synchronized truth surfaces: `BDC_APP_KNOWLEDGE`, AppHeader labels, status helpers, checkpoint/handoff/migration/overview docs, skill-measurement docs, golden cases, and live runtime data. Stale baseline SHAs, two Overview copies, a stale shipping spec, an incomplete route README, and mismatched Skills wording demonstrate that review discipline alone does not keep these surfaces aligned.

### Surface growth lacks an explicit complexity budget

Product/IA, Accessibility, Performance/Build, and Docs show the same growth from both sides. Users face twelve header destinations/utilities, three capture modes without a chooser, and help/chat interactions without consistent focus or live semantics. Maintainers carry 46 UI primitive files, 26 Radix packages with only four evidenced live consumers, duplicate Overview assets, an always-mounted HelpBubble, and two full builds without bundle or timing budgets. Surface growth is therefore both cognitive and operational, not merely visual.

## Ranked simplification proposals

These are proposals only. No follow-up issue was opened; the Projektleiter decides what becomes work.

### 1. Canonical Cockpit read projection

Create one non-protected, side-effect-free read projection that maps existing durable state into canonical display vocabulary, role labels, action reasons, lifecycle facets, and honest unknown values. UI routes, App Help facts, documentation tables, and contract tests should consume or be generated from that projection without changing pause, approval, auth, paths, or shipping authority.

**Supporting domains:** Product/IA, Shipping, Assistant, Auth, Docs, Tests.

**Why high leverage:** It removes multiple classes of contradictory copy and hidden state at once, while making future optimization safer than per-screen wording fixes.

### 2. Unknown-first UI state contract

Adopt one small route/query-state pattern that renders loading, error, empty, stale/sample, and success as mutually exclusive states with matching status/alert semantics and retry behavior. Apply it first to Plan, Done, Control connection status, Skills, and async chat/help updates.

**Supporting domains:** Product/IA, Shipping, Data, Tests, Accessibility.

**Why high leverage:** It repairs honesty, accessibility, and operational observability together; individual empty-state copy changes would solve only one symptom.

### 3. Executable boundary matrix

Turn existing boundary intent into caller-level table tests: anonymous/brother/owner server functions; BDC issue ownership before lifecycle writers; Q&A/rewrite/status contexts across model outputs; golden-helper/production-engine parity; GitHub/DB error contracts; and the existing path/guardrail negative matrix as the template. This proposal adds evidence around protected behavior rather than altering it.

**Supporting domains:** Auth, Shipping, Assistant, Tests, Data.

**Why high leverage:** It converts safety from import-topology knowledge into executable evidence and prevents green unit/eval suites from masking end-to-end gaps.

### 4. State ownership and truth index

Publish one maintained index that names, for each record and document: canonical source, mirror/cache, join key, retention/freshness, audience, and precedence. Cover GitHub issue/PR/events, engine comments, Neon and memory fallbacks, skill snapshots, learnings JSONL, checkpoint/handoff/migration/overview, route-role vocabulary, deployment identity, and validation evidence. Pair it with read-only integrity reporting for malformed/dropped records.

**Supporting domains:** Data, Docs, Product/IA, Shipping, Assistant, Tests.

**Why high leverage:** It reduces tribal knowledge and prevents stale or lossy mirrors from being mistaken for current truth without introducing a new database or migration.

### 5. Thin product shell and measurable build budget

Consolidate navigation utilities, label the three capture modes by purpose, preserve accessible focus/live behavior, and set explicit route/bundle/build-time budgets before adding more surfaces. Remove only evidence-backed dead or duplicated shipped assets after consumer checks.

**Supporting domains:** Product/IA, Accessibility, Performance/Build, Docs.

**Why high leverage:** It makes future features pay a visible complexity cost across user navigation, accessibility, bundle size, and verification time instead of accumulating independently.
